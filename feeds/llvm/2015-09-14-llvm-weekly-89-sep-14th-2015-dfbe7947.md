### On the mailing lists

- James Knight is proposing to [deprecate and remove the old SelectionDAG scheduler](http://lists.llvm.org/pipermail/llvm-dev/2015-September/090300.html), given that machine schedulers are now the preferred approach. He notes that a number of in-tree targets still use the SelectionDAG scheduler. It seems there is [support for this plan](http://lists.llvm.org/pipermail/llvm-dev/2015-September/090309.html).

- Jauhien is curious about the [availability of a C API for the ORC JIT](http://lists.llvm.org/pipermail/llvm-dev/2015-September/090120.html), with the motivating use case here being to provide a binding for Rust. The [main concern](http://lists.llvm.org/pipermail/llvm-dev/2015-September/090155.html) is that the ORC API is not yet stable, meaning it's not feasible to provide stable C bindings. The proposal is they live in llvm/include/llvm-c/unstable.

- Joseph Tremoulet has a whole bunch of questions about addrspacecast semantics, and Chandler Carruth has [a whole bunch of answers](http://lists.llvm.org/pipermail/llvm-dev/2015-September/090180.html).

- David Chisnall has a [useful response](http://lists.llvm.org/pipermail/llvm-dev/2015-September/090240.html) to a question about implementing LLVM intrinsics with multiple return values. As he points out, this is usually done by returning a struct.


### LLVM commits

- A major modification of LLVM'a alias analysis manager has landed in order to port it to the new pass manager. See the commit message for full details. [r247167](http://reviews.llvm.org/rL247167).

- The scalar replacement of aggregates (SROA) pass has been ported to the new pass manager. In the commit message, Chandler comments he hopes this serves as a good example of porting a transformation pass with non-trivial state to the new pass manager. [r247501](http://reviews.llvm.org/rL247501).

- The GlobalsModRef alias analysis pass is now enabled by default. [r247264](http://reviews.llvm.org/rL247264).

- Emacs users, rest your aching pinky fingers for a moment and rejoice. A range of improvements for the Emacs LLVM IR mode have landed. [r247281](http://reviews.llvm.org/rL247281).

- The AArch64 backend can now select STNP, the non-temporal store instruction (this hints that the value need not be kept in cache). [r247231](http://reviews.llvm.org/rL247231).

- Shrink wrapping optimisations are enabled on PPC64. [r247237](http://reviews.llvm.org/rL247237).

- A whole bunch of StringRef functions have been sprinkled with the `ALWAYS_INLINE` attribute so as to reduce the overhead of string operations even on debug LLVM builds. Chandler has also been making other changes to improve the performance of check-llvm with a debug build. [r247253](http://reviews.llvm.org/rL247253).

- The LLVM performance tips document has been extended to detail the use of allocas and when to specify alignment. [r247301](http://reviews.llvm.org/rL247301).

- The `hasLoadLinkedStoreConditional` TargetLoweringInformation callback has now been split in to `bool shouldExpandAtomicCmpXchgInIR(inst)` and `AtomicExpansionKind shouldExpandAtomicLoadInIR(inst)`. [r247429](http://reviews.llvm.org/rL247429).


### Clang commits

- A new control-flow integrity variant has been introduced, indirect function call chacking (enabled with `-fsanitize=cfi-icall`). This checks the dynamic type of the called function matches the static type used at the call. [r247238](http://reviews.llvm.org/rL247238).

- A new `-analyzer-config` option is available to modify the size of function that the inliner considers as large. [r247463](http://reviews.llvm.org/rL247463).

- Clang will now try much harder to preserve alignment information during IR-generation. [r246985](http://reviews.llvm.org/rL246985).

- The `__builtin_nontemporal_store` and `__builtin_nontemporal_load` builtins have been introduced. [r247104](http://reviews.llvm.org/rL247104), [r247374](http://reviews.llvm.org/rL247374).


### Other project commits

- libcxx gained implementations of Boyer-Moore and Boyer-Moore-Horspool searchers (for the language fundamentals technical specification). [r247036](http://reviews.llvm.org/rL247036).

- A trivial dynamic program linked with the new ELF lld now works with musl's dynamic linker. [r247290](http://reviews.llvm.org/rL247290).

- LLD's COFF linker learned to merge cyclic graphs, which means self-linking now produces a 27.7MB rather than a 29.0MB executable. MSVC manages to produce a 27.1MB executable, so there is still room for improvement. [r247387](http://reviews.llvm.org/rL247387).
